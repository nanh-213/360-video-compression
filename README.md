# 360° Video Compression & Streaming System

## Overview
This project serves as a simulation for a Viewport-Adaptive 360° Video Streaming System. While a fully deployed adaptive streaming protocol (like MPEG-DASH or HLS) saves actual network bandwidth by fetching video chunks on-demand, this application focuses on demonstrating the core encoding algorithms and client-side rendering logic. 

Instead of implementing a complex network transmission pipeline, the system generates and pre-loads two complete video streams to simulate how a player would handle geometric distortion and dynamically render the active viewport in real-time.

### Key Algorithms & Core Concepts
1. **Geometry-based Distortion Map (Equirectangular Projection):** Because 360° videos in Equirectangular format suffer from severe pixel stretching at the poles, the system calculates a latitude-based weight using the $cos(\text{latitude})$ function. This weight determines the Quantization Parameter Offset (Q-Offset) for different regions, assigning high quality to the equator and reducing redundant data at the poles.
2. **Simulated Adaptive ROI Encoding:** Utilizing `FFmpeg` and the `libx265` (HEVC) codec, the backend simultaneously encodes the original video into two distinct quality layers:
   - **HIGH Stream:** High base quality (CRF 20) with deep Q-Offset adjustments for the active viewport.
   - **LOW Stream:** Low base quality (CRF 32) representing the heavily compressed background vision.
3. **Raycasting & 3D Spatial Mapping:** The frontend uses `Three.js` to construct a 3D sphere divided into a 4x4 tile grid. A real-time **Raycasting algorithm** detects which tiles intersect with the camera's current viewing angle. It then dynamically swaps the textures of those specific tiles from the LOW video source to the HIGH video source.
4. **Real-time Metric Evaluation:** The system calculates and displays technical metrics including File Size, Bitrate, PSNR (Peak Signal-to-Noise Ratio), Compression Ratio, and Hardware Encoding Latency to evaluate the theoretical efficiency of the encoding strategy.

## Project Structure
```text
├── app.py                  # Backend Flask Server (handles FFmpeg encoding & math algorithms)
├── templates/
│   └── index.html          # Frontend UI, Three.js 3D rendering, and Dashboard logic
├── README.md               # Project documentation

```

> **Note:** Temporary directories (`uploads/` for input files and `static/output/` for processed chunks/heatmaps) are automatically generated by Python during runtime and are excluded from version control.

## System Requirements

To run this project locally, ensure you have the following installed:

1. Python 3.x
2. FFmpeg: Must be installed and added to your system's Environment Variables (PATH).
3. Python Packages:
```bash
pip install -r requirements.txt
```



---

## How to Run the Code

1. **Clone the repository** to your local machine:
```bash
git clone <your-github-repo-url>
cd <repository-folder>

```


2. **Start the Flask Server** by running the backend script:
```bash
python app.py

```


3. **Access the Web Interface**:
Open your web browser and navigate to:
```text
[http://127.0.0.1:5000](http://127.0.0.1:5000)

```


4. **Run the Live Demo**:
Upload any 360° panorama video (`.mp4` format) through the UI and click the start button. The system will process the Distortion Map, encode the streams, and initialize the 3D viewport environment.

## Important Notes

* **Hardware Utilization:** The real-time HEVC encoding process relies heavily on CPU performance. If the processing takes too long, try using a shorter 360° video (e.g., 5-10 seconds) for demonstration purposes.
* **Cross-Origin Video:** The video elements in the frontend are set with `crossorigin="anonymous"` to allow WebGL texture extraction without CORS issues.
